Нужно трансформировать существующий Ansible/Molecule проект.

Контекст:
- Сейчас в проекте есть только сценарий Molecule с Podman driver.
- Локально я хочу и дальше запускать тесты через Podman.
- Для GitHub Actions мне нужен отдельный сценарий, который не использует вложенные контейнеры, а применяет роль прямо к локальному хосту runner VM.
- GitHub runner эфемерный, поэтому деплой на localhost в CI допустим.
- Нужно максимально переиспользовать существующий код сценария и не дублировать playbook/verify/vars без необходимости.

Задача:
1. Проанализируй текущую структуру проекта и текущий Molecule scenario на Podman.
2. Добавь второй сценарий Molecule для GitHub Actions, который работает на localhost хоста раннера.
3. Сохрани текущий локальный сценарий Podman рабочим без регрессий.
4. Вынеси общий переиспользуемый код из текущего сценария туда, где им смогут пользоваться оба сценария.
5. Обнови документацию и CI-конфиги, чтобы было понятно, какой сценарий запускать локально, а какой в GitHub Actions.
6. Самостоятельно протестируй локальный Podman-сценарий после изменений и убедись, что он не сломан.

Что именно нужно сделать:
- Оставить существующий podman-сценарий для локального запуска.
- Добавить новый сценарий, например `gha-host` или аналогично понятное имя.
- Новый сценарий должен:
  - работать без вложенных контейнеров;
  - таргетить `localhost`;
  - использовать `ansible_connection: local`;
  - поддерживать `become: true`, если это нужно для роли;
  - не пытаться создавать/удалять контейнеры;
  - быть пригодным для запуска в GitHub Actions runner.
- Привести оба сценария к максимально общей структуре:
  - общий `converge.yml`, если это возможно;
  - общий `verify.yml`, если это возможно;
  - общий inventory/vars/defaults или иные общие части, если это разумно;
  - общие helper-файлы вынести из scenario-specific директорий.
- Избегать бессмысленного копипаста.
- Если проект уже использует ansible-native стиль Molecule, продолжай в этом стиле.
- Если проект пока в старом driver/platforms/provisioner стиле, не делай разрушительный рефакторинг ради красоты. Но если можно безопасно и компактно перейти на более чистую структуру без риска сломать текущий workflow — сделай это.
- Выбирай минимально инвазивный и поддерживаемый вариант.

Технические требования:
- Сценарий локального podman-запуска должен остаться удобным для `molecule test -s ...`.
- Сценарий для GitHub Actions должен запускаться отдельной командой `molecule test -s ...`.
- Общий playbook converge должен, по возможности, работать с `hosts: all`, чтобы один и тот же код применялся и к podman instance, и к localhost.
- Если verify-тесты завязаны на контейнерную среду, адаптируй их так, чтобы:
  - либо они были общими;
  - либо различия были минимальны и явны;
  - либо сценарий `gha-host` имел свою verify-часть только там, где это действительно нужно.
- Если в CI сейчас есть workflow под podman-сценарий, переключи GitHub Actions на новый host-based scenario.
- Локальный podman-сценарий не удалять.

Требования к валидации и проверке результата:
- После внесения изменений самостоятельно запусти существующий локальный podman-сценарий на текущей машине разработки.
- Не ограничивайся статическим анализом файлов: именно выполни сценарий Molecule локально.
- Минимум нужно прогнать команды, достаточные для проверки, что podman-сценарий по-прежнему работает после рефакторинга.
- Если полный `molecule test` слишком дорогой или медленный, как минимум выполни разумный набор из:
  - `molecule destroy -s <podman-scenario>`,
  - `molecule create -s <podman-scenario>`,
  - `molecule converge -s <podman-scenario>`,
  - `molecule idempotence -s <podman-scenario>`,
  - `molecule verify -s <podman-scenario>`,
  - и при необходимости финальный `molecule destroy -s <podman-scenario>`.
- Если есть возможность, предпочти полный `molecule test -s <podman-scenario>`.
- Зафиксируй фактический результат запуска:
  - какие команды были выполнены;
  - какие из них прошли успешно;
  - если были ошибки — что именно сломалось и как это исправлено.
- Не заявляй о сохранении работоспособности podman-сценария без фактического прогона.
- Если среда разработки не позволяет выполнить Podman/Molecule тест реально, явно напиши это, укажи точную причину и не утверждай, что сценарий проверен.
- После всех изменений отдельно убедись, что новый GHA-host scenario не ломает структуру существующего podman-сценария.
